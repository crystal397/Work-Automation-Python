## 1. 프로젝트 개요

### 1.1 배경 및 필요성

공공 건설공사의 공기연장 보고서를 작성할 때, 적용해야 하는 법령은 해당 계약의 입찰공고 시점에 따라 달라진다. 국가계약법, 지방계약법, 조달사업법, 하도급법, 산업안전보건관리비 기준 등은 매년 또는 수시로 개정되며, 법률·시행령·시행규칙이 각기 다른 시점에 개정되기 때문에, 입찰공고 시점 기준의 정확한 버전을 찾아내는 것이 핵심이다.

실무에서는 개정된 법령을 최신 버전으로만 참조하거나, 부칙의 경과규정을 확인하지 않고 시행일만으로 판단하여 잘못된 법령 버전을 인용하는 사례가 발생한다. 본 시스템은 입찰공고일을 입력받아 해당 시점에 실제로 시행 중이던 법령 버전을 자동으로 매칭하고, 부칙상 경과규정까지 검토하여 근거조문을 원문 그대로 제시함으로써, 보고서 작성 과정에서의 할루시네이션과 오적용을 방지하는 것을 목적으로 한다.

### 1.2 프로젝트 목표

- **정확성** — 입찰공고일 기준 시행 중인 법령 버전을 부칙 경과규정까지 반영하여 정확하게 매칭
- **무(無)할루시네이션** — 법령 조문은 AI가 생성하지 않고, 공식 API에서 받은 원문만 사용
- **추적가능성** — 모든 매칭 결과에 법령명, 공포번호, 시행일, 조문번호, 출처 URL을 함께 표시
- **편의성** — Python 스크립트 및 Windows exe 실행파일 형태로 제공하여 오프라인 환경에서도 활용 가능

## 2. 데이터 소스 — 법제처 API 경로 비교 (⚠ 결정 필요)

법제처 국가법령정보는 **두 개의 창구**를 통해 접근할 수 있다. 두 경로 모두 원천 데이터는 법제처에서 동일하게 제공되지만, 인증 방식과 인터페이스가 다르다. 본 프로젝트 착수 전에 반드시 한 경로를 선택해야 한다.

### 2.1 경로 A — 국가법령정보 공동활용 (법제처 직접)

| 항목 | 내용 |
|---|---|
| 제공기관 | 법제처 (직접 운영) |
| 포털 URL | https://open.law.go.kr |
| API 요청 URL | `http://www.law.go.kr/DRF/lawSearch.do` (목록)<br>`http://www.law.go.kr/DRF/lawService.do` (본문)<br>`http://www.law.go.kr/DRF/lawHistory.do` (연혁) |
| 인증 방식 | OC (기관코드) = 가입 시 이메일 ID |
| 승인 절차 | 신청 → 담당자 검토 → 1~2일 이내 승인 |
| 트래픽 제한 | 명시적 제한 문구는 없으나 과다 호출 시 제한 가능 |
| 상업적 이용 | 제한됨 (승인 필요) |
| 지원 범위 | 법령, 행정규칙, 자치법규, 판례, 헌재결정례, 법령해석례, 조약 등 **광범위** |
| 응답 형식 | XML, JSON |

### 2.2 경로 B — 공공데이터포털 (data.go.kr 경유)

| 항목 | 내용 |
|---|---|
| 제공기관 | 법제처 (공공데이터포털 경유) |
| 포털 URL | https://www.data.go.kr/data/15000115/openapi.do |
| API 요청 URL | `http://apis.data.go.kr/1170000/law/lawSearchList.do` |
| 인증 방식 | 서비스키 (공공데이터포털 발급) |
| 승인 절차 | **자동승인** (개발·운영 단계 모두) |
| 트래픽 제한 | 개발계정 10,000건/일, 운영계정은 활용사례 등록 시 증량 가능 |
| 상업적 이용 | 공공데이터법에 따름 (일반적으로 폭넓게 허용) |
| 지원 범위 | 법령 본문 조회, 맞춤형 법령 목록 조회, 행정규칙 본문 조회, 신구법 비교 등 |
| 응답 형식 | XML, JSON |

### 2.3 경로 선택 가이드

| 관점 | 추천 경로 |
|---|---|
| **승인 절차 간소화** | 경로 B (자동승인) |
| **기능 풍부함 (연혁 검색, 부칙 조회 등 세부 기능)** | 경로 A (가장 완전한 라인업) |
| **트래픽이 예측 가능하고 제한된 환경** | 경로 B (10,000건/일 명시) |
| **법령 외 자료(판례, 해석례 등) 확장 가능성** | 경로 A |
| **사내 배포 시 공공데이터포털 활용실적 등록** | 경로 B |

> **✅ 결정 완료 (2026-04-17)**: **경로 A (국가법령정보 공동활용)** 선택.
> 가입 신청 완료, API 정상 사용 중.

## 5. 할루시네이션 방지 원칙

본 시스템에서 가장 중요한 비기능 요구사항은 **법령 조문의 정확성**이다. 잘못된 조문 인용은 보고서의 신뢰도를 근본적으로 훼손하므로, 다음 원칙을 엄격히 적용한다.

### 5.1 핵심 원칙

| 원칙 | 항목 | 내용 |
|---|---|---|
| **P1** | 원문 직접 인용 | 법령 조문은 반드시 법제처 API에서 받은 원문 텍스트만 사용. LLM이나 사람이 조문을 '요약'·'재작성'하지 않는다 |
| **P2** | 버전 추적성 | 모든 조문 인용 시 [법령명, 공포일, 공포번호, 시행일, 조번호]의 5요소 기록 |
| **P3** | 결정 근거 기록 | 왜 그 버전이 선택되었는지(입찰공고일, 시행일 비교 결과, 부칙 경과규정 검토 결과)를 로그로 기록 |
| **P4** | 불확실성 표시 | 부칙 해석이 자동으로 확정되지 않는 경우 '사용자 확인 필요' 플래그 표시, 두 후보 버전 모두 제시 |
| **P5** | API 응답 검증 | API 응답의 공포번호·시행일자를 로컬 보유 파일명의 메타데이터와 교차 검증 |

### 5.2 시행일 판단 로직 (6단계)

단순히 '입찰공고일 ≥ 시행일'만 비교해서는 안 되며, 아래 단계를 거쳐 최종 적용 버전을 확정한다.

1. **연혁 조회** — 해당 법령의 모든 연혁 버전(공포번호, 공포일, 시행일) 수집
2. **1차 후보 선정** — 시행일 ≤ 입찰공고일 인 버전 중 시행일이 가장 최근인 버전을 1차 후보로 선정
3. **부칙 경과규정 확인** — 1차 후보의 부칙을 파싱하여 두 가지 유형의 경과규정을 각각 탐지
   - **유형 A — 법령 전체 경과규정**: "이 법(령) 시행 전에 공고(또는 계약 체결)된 건에 대하여는 종전의 규정에 따른다" → 전체 법령에 이전 버전 적용
   - **유형 B — 조문 단위 경과규정**: "제○조의 개정규정은 이 영 시행 후 최초로 입찰공고하는 분부터 적용한다" → 해당 조문만 이전 버전 적용, 나머지 조문은 신규 버전 적용
4. **2차 후보 선정** — 유형 A이면 전체 이전 버전으로 재선정. 유형 B이면 조문별 적용 버전이 달라지므로 해당 조문에 한해 직전 버전을 병기하고 사용자 확인 요청
5. **상위·하위법 정합성 확인** — 법률 버전과 시행령·시행규칙 버전의 시행일이 서로 일치하거나 논리적으로 정합한지 확인, 불일치 시 경고
6. **사용자 검토 요청** — 자동 판단이 어려운 경우(부칙 해석이 모호한 경우 등) 두 후보 버전을 모두 출력하고 사용자가 선택

> **⚠️ 경과규정 패턴 예시**
>
> 국가계약법 부칙에 자주 등장하는 경과규정 표현:
> - "이 영 시행 전에 입찰공고한 계약에 관하여는 종전의 규정에 따른다."
> - "이 법 시행 당시 종전의 규정에 따라 체결된 계약에 관하여는 종전의 규정에 따른다."
> - "제○조의 개정규정은 이 영 시행 후 최초로 입찰공고하는 분부터 적용한다."
>
> 이러한 문장 패턴을 정규표현식 또는 키워드 기반으로 탐지하고, 탐지된 경우 **사용자에게 반드시 알린다**.

> **🔄 비정형 패턴 탐지 — Claude Code CLI fallback**
>
> 정규표현식으로 탐지되지 않는 비정형 경과규정 문장의 경우, `claude -p "..."` CLI를 subprocess로 호출하여 A/B 유형 분류 및 영향 조번호 추출을 시도한다. Claude CLI가 설치되지 않은 환경에서는 조용히 스킵하며, 탐지 실패 시 P4 원칙에 따라 두 후보 버전을 모두 출력한다.

## 6. 기술 요구사항

### 6.1 개발 환경

| 항목 | 내용 |
|---|---|
| 언어 | Python 3.10 이상 |
| 주요 라이브러리 | `requests` (API 호출), `xmltodict` (XML 파싱), `sqlite3` (캐시, 표준 라이브러리), `python-docx` (Word 리포트), `python-dotenv` (`.env` 로드), `customtkinter>=5.2.0` (모던 GUI 프레임워크), `tkinter` (표준 라이브러리 — filedialog·messagebox·ttk 사용), `tkcalendar` (날짜 선택 달력), `beautifulsoup4` (admrul 연혁 웹 스크래핑), `selenium>=4.6.0` (JS 렌더링 페이지 스크래핑 — Selenium Manager가 chromedriver 자동 관리), `pyinstaller` (exe 빌드) |
| 패키징 | PyInstaller로 Windows용 단일 exe 빌드 (`--onefile` 옵션) |
| 외부 API | 법제처 국가법령정보 (§2에서 경로 확정) |
| 대상 OS | Windows 10 / 11 (우선), macOS는 Python 스크립트로 실행 |

### 6.2 시스템 구조

| 계층 | 역할 |
|---|---|
| ① GUI Layer | CustomTkinter 기반 모던 GUI — 입찰공고일 입력, 체크박스 법령 선택, 결과 탭 표시, 경과규정 경고 색상 강조 |
| ② Engine Layer | 시행일 기준 버전 선정, 부칙 경과규정 탐지, 조문 필터링 등 핵심 비즈니스 로직 |
| ③ API Client Layer | 법제처 OPEN API 호출, XML 파싱, 에러 처리, 재시도 로직 |
| ④ Cache Layer | SQLite 기반 로컬 캐시 (법령 메타데이터, 연혁, 조문 본문), WAL 모드 |
| ⑤ Report Layer | Word(.docx) 형식의 결과 리포트 생성 |
| ⑥ Scraper Layer | 행정규칙(admrul) 연혁 웹 스크래핑 — requests/BS4 1단계, Selenium headless 2단계 (법령 law 타입은 API 실패 시 현행 버전 fallback) |

### 6.3 비기능 요구사항

- **성능** — 1개 법령 조회 시 API 응답 포함 5초 이내, 캐시 활용 시 1초 이내
- **신뢰성** — API 장애 시 로컬 캐시로 폴백, 네트워크 오류 시 최대 3회 재시도
- **로깅** — 모든 API 호출, 버전 선정 과정, 사용자 선택을 로그 파일에 기록 (감사 추적 목적)
- **보안** — API 키(OC)는 환경변수(`.env`) 또는 GUI 입력으로 관리, 소스코드 하드코딩 금지
